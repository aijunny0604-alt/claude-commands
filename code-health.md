# 코드 품질 점검 에이전트 팀 - PDCA 기술 부채 관리

당신은 코드 품질 전문 에이전트 팀 리더입니다. 3개 전문 에이전트 팀을 **동시에** 출동시켜 PDCA 사이클로 코드 품질 점검 → 리팩토링 → 재점검을 **90점 이상**까지 자동 반복합니다.

인자: $ARGUMENTS (점검 대상: "전체" 또는 특정 디렉토리/기능)

---

## PDCA 사이클 개요

```
Plan(코드 현황) → Do(3팀 동시 분석) → Check(점수화) → Act(리팩토링)
    ↑                                                     |
    └──────────── 90점 미만이면 반복 (최대 3회) ──────────┘
```

---

## Phase 1: PLAN (점검 계획)

### 1-1. 코드 현황 파악
```bash
# 전체 코드량
find src -name "*.ts" -o -name "*.tsx" | xargs wc -l | tail -1
# 파일 수
find src -name "*.ts" -o -name "*.tsx" | wc -l
# 최근 변경이 많은 파일 (핫스팟)
git log --pretty=format: --name-only --since="1 month ago" | sort | uniq -c | sort -rn | head -20
```

### 1-2. 디렉토리 구조 분석
- 컴포넌트 수, API 수, 유틸리티 수
- 파일 평균 줄 수, 최대 줄 수

### 1-3. 계획서 저장
파일: `docs/03-analysis/code-health-plan-{날짜}.md`

## Phase 2: DO (3개 에이전트 팀 동시 출동)

### 팀 1: 중복 코드 탐지반 (code-analyzer 에이전트)

**코드 중복 + 공통화 기회 분석**

- **복붙 코드 탐지**:
  - 10줄 이상 동일/유사 코드 블록
  - 유사 컴포넌트 (같은 구조, 다른 데이터)
  - 유사 API 핸들러 (같은 CRUD 패턴 반복)
  - 유사 유틸 함수 (기능 중복)
- **공통 추출 가능 로직**:
  - API 에러 핸들링 패턴 → 공통 미들웨어
  - Prisma 쿼리 패턴 → 공통 서비스 레이어
  - 폼 처리 패턴 → 커스텀 훅
  - 모달 패턴 → 공통 모달 컴포넌트
- **일관성 위반**:
  - 같은 역할인데 다른 네이밍 (handle vs on, data vs result)
  - 같은 패턴인데 다른 구현 (fetch vs axios, try/catch vs .catch)
  - 파일 구조 불일치 (어떤 API는 service 분리, 어떤 건 인라인)

파일:줄번호 + 중복률(%) 포함.

### 팀 2: 복잡도 분석반 (Explore 에이전트)

**코드 복잡도 + 구조 분석**

- **함수/컴포넌트 길이**:
  - 100줄 이상 함수 → 분리 권장
  - 300줄 이상 컴포넌트 → God Component 경고
  - 500줄 이상 파일 → 파일 분리 권장
- **의존성 깊이**:
  - import 10개 이상 파일 → 역할 과다
  - 순환 참조 (A→B→A)
  - 깊은 컴포넌트 트리 (5단계+ prop drilling)
- **조건 복잡도**:
  - 중첩 if/else 3단계 이상
  - 삼항 연산자 중첩
  - switch 10개 이상 case
- **상태 관리 복잡도**:
  - useState 10개 이상인 컴포넌트
  - useEffect 5개 이상인 컴포넌트
  - 의존성 배열이 5개 이상인 useEffect
- **매직 넘버/스트링**:
  - 하드코딩된 숫자 (상수화 필요)
  - 반복되는 문자열 (enum/const화 필요)

파일:줄번호 + 복잡도 수치 포함.

### 팀 3: 미사용 코드 + 의존성 점검반 (general-purpose 에이전트)

**Dead Code + 의존성 분석**

- **미사용 코드**:
  - export 했지만 어디서도 import 안 되는 함수/컴포넌트
  - 정의했지만 호출 안 되는 함수
  - 주석 처리된 코드 블록 (삭제 권장)
  - TODO/FIXME/HACK 주석 목록화
- **미사용 파일**:
  - 어디서도 import 안 되는 파일
  - 빈 파일, 보일러플레이트만 있는 파일
- **의존성 점검**:
  ```bash
  # 미사용 패키지 탐지
  npx depcheck 2>/dev/null || echo "depcheck not installed"
  # 패키지 크기 확인
  cat package.json | node -e "const p=require('fs').readFileSync('/dev/stdin','utf8');const j=JSON.parse(p);console.log('deps:',Object.keys(j.dependencies||{}).length,'devDeps:',Object.keys(j.devDependencies||{}).length)"
  ```
  - package.json에 있지만 코드에서 미사용 패키지
  - 중복 기능 패키지 (예: moment + dayjs)
  - 오래된 패키지 (major 버전 2개+ 뒤처짐)
- **TypeScript 품질**:
  - any 타입 사용 횟수
  - as 타입 캐스팅 남용
  - @ts-ignore / @ts-nocheck 사용

목록화 + 삭제 시 영향도 분석 포함.

## Phase 3: CHECK (점수화)

### 3-1. 3팀 결과 종합 → 코드 건강 점수 (100점)
| 감점 기준 | 점수 |
|----------|------|
| ❌ Critical (순환참조, God Component 500줄+, 심각한 중복) 1건당 | -15점 |
| ⚠️ High (100줄+ 함수, 10+ useState, 미사용 export) 1건당 | -8점 |
| 🔶 Medium (매직넘버, 일관성 위반, TODO) 1건당 | -3점 |
| 💡 Low (개선 제안) 1건당 | -1점 |

### 3-2. 분석 결과 저장
파일: `docs/03-analysis/code-health-result-{날짜}.md`

```
## 코드 건강 점수: XX/100

### 파일별 건강도
| 파일 | 줄수 | 복잡도 | 중복 | 미사용 | 상태 |
|------|------|:------:|:----:|:------:|:----:|
| App.tsx | 450 | ⚠️ | ✅ | ✅ | 보통 |

### 핫스팟 (가장 문제 많은 파일 Top 5)
1. 파일명 — 문제 N건 — 총 감점 X점
```

### 3-3. 판정
- **90점 이상**: Phase 5(보고서)
- **90점 미만**: Phase 4(Act)

## Phase 4: ACT (리팩토링 + 재점검)

**최대 3회 반복. 사용자 확인 후 진행.**

### 4-1. 자동 수정 우선순위
1. **미사용 코드 삭제** (안전, 사이드이펙트 없음)
2. **매직 넘버 상수화** (안전, 가독성 개선)
3. **중복 코드 공통화** (중위험, 테스트 필요)
4. **복잡 함수 분리** (중위험, 테스트 필요)

### 4-2. 빌드 확인
```bash
npx next build
```

### 4-3. 커밋
```bash
git commit -m "refactor: 코드 품질 개선 (PDCA Act #{N})"
```

### 4-4. 재점검 → 90점 이상이면 Phase 5

## 핵심 규칙

1. **PDCA 자동 반복**: 90점 미만 시 리팩토링 → 재점검 최대 3회
2. **안전한 수정부터**: 미사용 삭제 → 상수화 → 공통화 → 분리
3. **기능 유지**: 리팩토링이 기능을 깨뜨리면 안 됨
4. **빌드 테스트** 필수
5. **최소 변경**: 한 번에 너무 많이 바꾸지 않기 (단계적 개선)
6. **문서화**: 계획서, 분석, 보고서 모두 docs/ 저장
