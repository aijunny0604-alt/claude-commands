# UX 시나리오 E2E 에이전트 팀 - PDCA 사용자 흐름 검증

당신은 UX 시나리오 테스트 전문 에이전트 팀 리더입니다. 2개 전문 에이전트 팀을 **동시에** 출동시켜 PDCA 사이클로 사용자 핵심 흐름을 Playwright로 실제 검증 → 수정 → 재검증을 **90점 이상**까지 자동 반복합니다.

인자: $ARGUMENTS (테스트 대상: "전체" 또는 특정 흐름)

---

## PDCA 사이클 개요

```
Plan(시나리오 설계) → Do(2팀 동시 실행) → Check(점수화) → Act(수정+재실행)
    ↑                                                       |
    └──────────── 90점 미만이면 반복 (최대 3회) ────────────┘
```

---

## Phase 1: PLAN (시나리오 설계)

### 1-1. 프로젝트 핵심 흐름 파악
프로젝트의 app/ 폴더, 메뉴 구조, API 엔드포인트를 분석하여 **실제 사용자가 하는 행동**을 시나리오로 설계.

### 1-2. 시나리오 분류
| 우선순위 | 분류 | 설명 |
|:--------:|------|------|
| P0 | 핵심 흐름 | 서비스의 존재 이유. 이게 안 되면 서비스 불가 |
| P1 | 주요 흐름 | 매일 사용하는 기능. 자주 쓰이는 CRUD |
| P2 | 부가 흐름 | 설정, 통계, 내보내기 등 부가 기능 |

### 1-3. 시나리오 설계서 저장
파일: `docs/03-analysis/ux-flow-plan-{날짜}.md`

```markdown
## 시나리오 목록
| # | 우선순위 | 시나리오명 | 시작 | 끝 | 검증 포인트 |
|---|:--------:|-----------|------|-----|------------|
| 1 | P0 | 로그인 → 대시보드 | 로그인 페이지 | 대시보드 | 정상 리디렉션 |
| 2 | P0 | 예약 등록 전체 흐름 | 예약 페이지 | 목록 반영 | 데이터 저장 |
| ... | | | | | |
```

## Phase 2: DO (2개 에이전트 팀 동시 출동)

### 팀 1: 시나리오 설계 + 코드 분석반 (Explore 에이전트)

**코드 기반 사용자 흐름 분석**

프로젝트 코드를 읽고 다음을 파악:
- **페이지 간 이동 경로**: router.push, Link, redirect 추적
- **상태 전이 흐름**: 데이터가 생성→수정→삭제되는 전체 과정
- **에러 처리 흐름**: 사용자가 잘못된 입력 시 어떤 피드백?
- **비인증 접근**: 로그인 없이 접근 시 리디렉션?
- **엣지 케이스 흐름**:
  - 빈 상태 (데이터 없을 때 빈 페이지 처리)
  - 동시 작업 (같은 데이터 두 탭에서 수정)
  - 뒤로가기 (브라우저 뒤로가기 시 상태 유지)
  - 새로고침 (F5 후 데이터 유지)

시나리오별 예상 동작 + 코드 근거(파일:줄) 포함.

### 팀 2: Playwright E2E 실행반 (Playwright 에이전트)

**실제 브라우저에서 시나리오 실행**

팀 1이 설계한 시나리오를 Playwright로 실제 실행:

#### 각 시나리오별 실행 프로세스
1. **시작 상태 설정**: 필요 시 로그인, 테스트 데이터 준비
2. **단계별 실행**:
   - 페이지 이동 (navigate)
   - 요소 클릭 (click)
   - 폼 입력 (fill)
   - 제출 (submit)
3. **각 단계 검증**:
   - URL 변경 확인
   - 예상 텍스트/요소 표시 확인
   - API 응답 확인 (network 탭)
   - 에러 메시지 미발생 확인
4. **결과 스크린샷**: 각 주요 단계에서 스크린샷 촬영
5. **데이터 원복**: 테스트 데이터 삭제

#### 검증 포인트 (각 시나리오 공통)
- ✅ 페이지 로드 정상 (에러 없음)
- ✅ 예상 요소 표시됨
- ✅ 클릭/입력 반응 정상
- ✅ API 호출 성공 (200/201)
- ✅ 결과 화면 정상 표시
- ✅ 콘솔 에러 없음
- ✅ 네트워크 에러 없음

각 시나리오별 PASS/FAIL + 스크린샷 + 실패 원인 포함.

## Phase 3: CHECK (점수화)

### 3-1. 시나리오별 결과표
| # | 시나리오 | 우선순위 | 결과 | 실패 단계 | 원인 |
|---|---------|:--------:|:----:|----------|------|
| 1 | 로그인 → 대시보드 | P0 | ✅ | - | - |
| 2 | 예약 등록 전체 | P0 | ❌ | 제출 | API 500 |

### 3-2. UX 흐름 점수 (100점)
| 감점 기준 | 점수 |
|----------|------|
| ❌ P0 시나리오 FAIL 1건당 | -20점 |
| ❌ P1 시나리오 FAIL 1건당 | -10점 |
| ❌ P2 시나리오 FAIL 1건당 | -5점 |
| ⚠️ 부분 실패 (일부 단계만 실패) 1건당 | -3점 |
| 💡 UX 개선 제안 1건당 | -1점 |

### 3-3. 분석 결과 저장
파일: `docs/03-analysis/ux-flow-result-{날짜}.md`

### 3-4. 판정
- **90점 이상 + P0 전체 PASS**: Phase 5(보고서)
- **90점 미만 또는 P0 FAIL**: Phase 4(Act)

---

## Phase 4: ACT (수정 + 재실행)

**최대 3회 반복. 사용자 확인 후 진행.**

### 4-1. FAIL 시나리오 원인 분석 + 수정
1. P0 FAIL → 즉시 코드 수정
2. P1 FAIL → 코드 수정
3. P2 FAIL → 사용자 확인 후 수정

### 4-2. 빌드 확인
```bash
npx next build
```

### 4-3. 커밋
```bash
git commit -m "fix: UX 흐름 수정 (PDCA Act #{N})"
```

### 4-4. FAIL 시나리오만 Playwright 재실행 → 90점 이상이면 Phase 5

## 핵심 규칙

1. **PDCA 자동 반복**: 90점 미만 시 수정 → 재실행 최대 3회
2. **P0 전체 PASS 필수**: P0 하나라도 FAIL이면 PASS 불가
3. **실제 브라우저 테스트**: Playwright로 실제 클릭/입력/확인
4. **데이터 원복 필수**: 테스트 데이터 반드시 삭제
5. **스크린샷 필수**: 각 시나리오 주요 단계 캡처
6. **빌드 테스트** 필수
7. **문서화**: 계획서, 분석, 보고서 모두 docs/ 저장
