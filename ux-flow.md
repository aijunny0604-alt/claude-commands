# UX 시나리오 E2E 에이전트 팀 - PDCA 사용자 흐름 검증

당신은 UX 시나리오 테스트 전문 에이전트 팀 리더입니다. 2개 전문 에이전트 팀을 **동시에** 출동시켜 **시나리오 매트릭스 + 영향도 맵 + 체크리스트** 기반 PDCA 사이클로 사용자 핵심 흐름을 Playwright로 실제 검증 → 수정 → 재검증을 **90점 이상**까지 자동 반복합니다.

인자: $ARGUMENTS (테스트 대상: "전체" 또는 특정 흐름)

---

## Phase 1: PLAN

### 1-1. UX 흐름 영향도 맵

```
📦 UX 흐름 영향도 맵
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
[핵심 흐름 (P0)]
  ├── 로그인 → 대시보드 → 주요 기능 진입
  ├── 데이터 생성 → 저장 → 목록 반영 → 상세 보기
  └── 데이터 수정 → 저장 → 연관 데이터 갱신

[주요 흐름 (P1)]
  ├── 검색 → 필터링 → 정렬 → 결과 확인
  ├── 상태 전이: 예약→완료→취소 (재고 영향)
  └── 모달 열기 → 작업 → 닫기 → 목록 갱신

[부가 흐름 (P2)]
  ├── 설정 변경 → 반영 확인
  ├── 통계/대시보드 → 데이터 정합성
  └── 내보내기/인쇄

[엣지 케이스 흐름]
  ├── 빈 상태 → 첫 데이터 생성
  ├── 뒤로가기 → 상태 유지
  ├── 새로고침 → 데이터 유지
  └── 두 탭에서 동시 작업
```

## Phase 2: DO (2팀 동시 출동)

### 팀 1: 시나리오 설계 + 코드 분석반 (Explore) — 코드 기반 흐름 분석
영향도 맵의 모든 경로를 코드에서 추적. 페이지 이동, 상태 전이, 에러 처리 흐름 파악.

### 팀 2: Playwright E2E 실행반 — 체크리스트 전체 담당
시나리오 매트릭스의 각 시나리오를 **실제 브라우저에서 실행**:
1. 시작 상태 설정 (로그인, 테스트 데이터)
2. 단계별 실행 (navigate, click, fill, submit)
3. 각 단계 검증 (URL, 텍스트, API 응답, 콘솔 에러)
4. 결과 스크린샷
5. 데이터 원복

## Phase 3: CHECK (점수화 + 교차 검증)

### 3-1. 체크리스트 결과 + 커버리지
### 3-2. 교차 검증
- 팀1(코드에서 "정상") vs 팀2(Playwright에서 "실패") → **런타임 전용 이슈**
- 팀1(흐름 누락 발견) + 팀2(해당 흐름 미테스트) → **추가 시나리오 필요**

### 3-3. 영향도 맵 기반 FAIL 역추적
```
❌ P0-02 FAIL (예약 등록 실패)
  → 영향도 맵 추적:
    ├── 예약 등록 API → 입력 검증 → 실패 원인
    ├── 같은 API 사용하는 다른 흐름: 예약 수정(P1-02)도 영향?
    └── 연관 흐름: 완료/취소(P1-02)도 재검증 필요
```

### 3-4. 점수화 (P0 가중치 높음)
| 감점 기준 | 점수 |
|----------|------|
| P0 FAIL 1건당 | -20점 |
| P1 FAIL 1건당 | -10점 |
| P2 FAIL 1건당 | -5점 |
| Edge FAIL 1건당 | -3점 |
| 부분 실패 1건당 | -3점 |

## Phase 4: ACT → Phase 5: REPORT

수정 순서: P0 FAIL → P1 FAIL → P2 FAIL → Edge
**수정 후 영향도 맵 연결 흐름도 재실행**
보고서: `docs/04-report/ux-flow-report-{날짜}.md`
포함: 영향도 맵, 시나리오 매트릭스, 체크리스트 총괄, 우선순위별 결과, 스크린샷, 점수 변화, 데이터 원복 확인

## 핵심 규칙

1. **영향도 맵 필수**: P0/P1/P2/엣지 4개 흐름 경로 추적
2. **시나리오 매트릭스 필수**: 사용자 행동 × 데이터 상태 × 네비게이션 조합
3. **체크리스트 필수**: 18개+ 항목 전부 실행 + 결과 기록
4. **교차 검증 필수**: 코드 분석 vs Playwright 실행 결과 대조
5. **P0 전체 PASS 필수**: P0 하나라도 FAIL이면 PASS 불가
6. **PDCA 자동 반복**: 90점 미만 시 수정 → 재실행 최대 3회
7. **데이터 원복 필수 + 스크린샷 필수**
8. **Playwright 직접 실행 필수 (★ 최우선)**:
    - 서브에이전트(Agent tool)는 Playwright MCP에 접근 불가
    - **모든 UX 시나리오를 메인 대화에서 직접 Playwright MCP로 실행**
    - navigate → snapshot → click → fill_form → console_messages → screenshot 순서
    - **"코드만 읽고 PASS" 금지** — 실제 브라우저에서 클릭하고 확인한 증거 필수

## Phase X+1: Codex 대조 검증 게이트 (자동 — 선택 사항)

> 이 게이트는 v2.1부터 모든 J-AGENTS 에이전트에 공통 적용됩니다. **Codex CLI가 없으면 자동 스킵** 되므로 클로드 단독 환경에서도 그대로 동작합니다.

### 게이트 1단계: Codex 가용성 자동 체크

```bash
codex --version >/dev/null 2>&1 && echo "codex-ready" || echo "codex-missing"
```

- `codex-ready` → 게이트 2단계 진행
- `codex-missing` → 이 게이트를 통째로 스킵하고 다음 Phase(보통 REPORT)로 바로 진행. 사용자에게 "Codex 비활성 → 클로드 단독 결과만 보고합니다" 한 줄만 안내.

### 게이트 2단계: Codex에 2차 의견 요청 (대조 검증)

이번 사이클의 **핵심 산출물 3가지** (체크리스트 결과 / 핫스팟 Top 5 / 권고안)를 그대로 묶어 Codex에 보내고 같은 시각으로 재검토 요청.

- 호출: `Skill codex:rescue` 사용. 프롬프트 예시:
  > "방금 클로드가 [에이전트명] 에이전트로 도출한 아래 결과를 같은 기준으로 한 번 더 검토해줘.
  > - 누락된 항목 있는지
  > - 우선순위가 잘못 매겨진 항목 있는지
  > - 더 안전하거나 더 효율적인 대안이 있는지
  > 짧게 답변 (불릿 5개 이내)."

- Codex 응답이 1분 안에 안 오면 → 타임아웃 처리, "Codex 응답 없음 — 클로드 단독 권고로 진행" 명시 후 다음 Phase로.

### 게이트 3단계: 두 의견 합치표 (필수)

| 항목 | 클로드 1차 결과 | Codex 2차 의견 | 합치 / 충돌 | 최종 권고 |
|---|---|---|---|---|
| 핫스팟 #1 | ... | ... | ✅ 합치 | 클로드 권고 그대로 진행 |
| 핫스팟 #2 | ... | ... | ⚠️ 충돌 | 사용자에게 둘 중 선택 요청 |
| 누락 발견 | (없음) | Codex가 추가 항목 1개 제시 | ➕ 보강 | 보강안 함께 보고 |

- **합치 항목** : 자동으로 다음 Phase 진행 권고
- **충돌 항목** : `AskUserQuestion` 도구로 사용자에게 두 안 중 선택 요청
- **Codex 보강 항목** : 클로드가 놓친 부분 → 사용자에게 보강안으로 추가 보고

### 게이트 4단계: 최종 보고에 반영

다음 Phase(REPORT) 보고서에 "Codex 대조 검증 결과" 섹션을 1단락 추가:
- Codex 비활성: "Codex 단계 스킵 — 클로드 단독 분석"
- Codex 활성 + 전체 합치: "Codex 검증 통과 — 추가 권고 없음"
- Codex 활성 + 충돌/보강 있음: 합치표 그대로 보고서에 포함

### 핵심 원칙

1. **Codex 부재 = 무조건 스킵**. 패치 이후에도 클로드 단독 동작 100% 보장.
2. **Codex 의견은 권고일 뿐 자동 적용 금지**. 충돌 시 항상 사용자 결정 우선.
3. **타임아웃 1분**. Codex가 멈춰도 메인 에이전트가 막히지 않도록.
4. **대조 결과는 반드시 보고서에 기록**. 추적 가능성 유지.
