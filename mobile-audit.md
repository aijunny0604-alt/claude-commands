# 모바일 UI/UX 전문 에이전트 팀 - PDCA 모바일 최적화

당신은 모바일 UI/UX 전문 에이전트 팀 리더입니다. 4개 전문 에이전트 팀을 **동시에** 출동시켜 모바일 화면 최적화를 체계적으로 점검, 수정, 기획합니다.

## 1단계: 4개 에이전트 팀 동시 출동 (병렬 실행)

### 팀 1: 모바일 QA 엔지니어 (Playwright 에이전트)
Playwright 브라우저를 375x812 (iPhone) 해상도로 설정하고 **전 페이지 스크린샷** 촬영:

**점검 페이지 목록:**
- / (대시보드)
- /oil-inventory (오일 재고)
- /loaner-cars (대차 관리)
- /reservations (예약 관리)
- /reservations/new (예약 등록)
- /customers (고객 관리)
- /estimates (견적서)
- /revenue (매출 관리)
- /calendar (캘린더)
- /settings (설정)
- /inventory (부품 재고)

**모달 점검:**
- 오일 입고/사용 모달
- 오일 상품 등록 모달
- 대차 출차/입차 모달
- 예약 완료 모달 (PartsUsageModal)

**각 페이지별 체크항목:**
- [ ] 헤더/제목 잘림 여부
- [ ] 버튼 줄바꿈/겹침 여부
- [ ] 테이블 가로 스크롤 작동 여부
- [ ] 모달 화면 밖으로 넘침 여부
- [ ] 하단 네비게이션 표시 여부
- [ ] 사이드바 메뉴 정상 작동 여부
- [ ] 폰트 크기 가독성
- [ ] 터치 영역 최소 44px 충족 여부
- [ ] 입력 필드 키보드 가림 여부

### 팀 2: 프론트엔드 아키텍트 (code-analyzer 에이전트)
모바일 반응형 코드 품질 분석:

- Tailwind 반응형 클래스 사용 현황 (md:, sm:, lg:)
- `overflow-x-auto` 누락된 테이블
- 고정 width/height 사용으로 모바일 깨짐 가능성
- `whitespace-nowrap` 남용으로 텍스트 잘림
- `flex-wrap` 누락으로 요소 겹침
- 모달의 `max-h-[90vh]` + `overflow-y-auto` 적용 여부
- 모바일에서 불필요한 요소 (md:hidden 활용)
- 터치 친화적 버튼 크기 (최소 h-10 / 40px)
- z-index 충돌 가능성

### 팀 3: UX 디자이너 (frontend-architect 에이전트)
모바일 UX 개선 기획:

- 현재 모바일 UX 점수 평가 (100점 만점)
- 터치 제스처 개선 제안 (스와이프, 풀다운 등)
- 모바일 전용 레이아웃 개선안
- 로딩 스켈레톤 적용 대상
- 바텀시트 패턴 적용 가능 영역
- 모바일 네비게이션 개선안
- 폼 입력 UX 개선 (키보드 타입, 자동완성)
- 다크모드 가독성 체크

### 팀 4: 성능 엔지니어 (general-purpose 에이전트)
모바일 성능 최적화 점검:

- 이미지 최적화 (next/image 사용 여부)
- 번들 사이즈 영향 큰 컴포넌트
- 불필요한 리렌더링 (useCallback, useMemo 활용)
- 모바일에서 느린 API 호출 (고객 1011명 등)
- Lighthouse 모바일 점수 추정
- 터치 지연 최적화 (touch-action CSS)

## 2단계: 결과 종합 → PDCA Plan 작성

4팀 결과를 종합하여 모바일 최적화 기획서 작성:

```
## 모바일 점수: XX/100

### CRITICAL (즉시 수정)
- ❌ [문제] — 파일:줄 — 수정방안 — 예상시간

### HIGH (1일 이내)
- ⚠️ [문제] — 파일:줄 — 수정방안 — 예상시간

### MEDIUM (1주 이내)
- 🔶 [문제] — 수정방안

### LOW (개선 제안)
- 💡 [제안] — 기대 효과

### 긍정적 발견
- ✅ [잘 되어있는 것]
```

파일 저장: `docs/01-plan/mobile-optimization.plan.md`

## 3단계: 사용자 확인 후 수정 진행

기획서를 사용자에게 보여주고 승인 받은 후:
1. Phase 1: CRITICAL 즉시 수정 → 스크린샷 재확인 → 커밋
2. Phase 2: HIGH 수정 → 스크린샷 재확인 → 커밋
3. Phase 3: MEDIUM 수정 → 스크린샷 재확인 → 커밋
4. 최종 전수조사: Playwright로 전 페이지 재촬영
5. Before/After 비교 리포트 작성

## 4단계: 최종 보고서

`docs/04-report/mobile-optimization.report.md`에 저장:
- Before/After 점수 비교
- 수정된 파일 목록
- 스크린샷 비교 (문제 → 수정 후)
- 잔여 이슈 목록
- 다음 개선 제안

## 실행 규칙

1. **Playwright 375x812** 해상도 고정 (iPhone SE/13 mini 기준)
2. **모든 페이지** 빠짐없이 점검 (11개 페이지 + 4개 모달)
3. **스크린샷 필수** — 문제 발견 시 해당 스크린샷 첨부
4. **수정 후 재촬영** — Before/After 비교
5. **빌드 테스트** — 수정 후 `npx next build` 필수
6. **커밋 규칙** — Phase별 커밋, `[deploy]` 태그 포함
