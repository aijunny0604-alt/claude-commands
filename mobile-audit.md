# 모바일 UI/UX 전문 에이전트 팀 - PDCA 모바일 최적화

당신은 모바일 UI/UX 전문 에이전트 팀 리더입니다. 4개 전문 에이전트 팀을 **동시에** 출동시켜 **시나리오 매트릭스 + 영향도 맵 + 체크리스트** 기반 PDCA 사이클로 모바일 점검 → 수정 → 재점검을 **90점 이상**까지 자동 반복합니다.

인자: $ARGUMENTS (점검 대상: "전체" 또는 특정 페이지)

---

## PDCA 사이클 개요

```
Plan(영향도맵+시나리오) → Do(4팀 동시 점검) → Check(점수+교차검증) → Act(수정+재촬영)
    ↑                                                                    |
    └──────────── 90점 미만이면 반복 (최대 3회) ────────────────────────┘
```

---

## Phase 1: PLAN (점검 계획 + 영향도 맵)

### 1-1. 점검 대상 파악
프로젝트의 모든 페이지와 모달을 자동 탐지 (app/ 폴더 + 모달 컴포넌트).

### 1-2. 모바일 UI 영향도 맵

```
📦 모바일 UI 영향도 맵
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
[레이아웃 경로]
  ├── 페이지 레이아웃 (flex/grid) → 모바일 축소 시 변화
  ├── 사이드바/네비 → 모바일 전환 방식
  └── 테이블 → 가로 스크롤 or 카드 전환

[컴포넌트 경로]
  ├── 모달 → max-height + overflow 관리
  ├── 폼 입력 → 키보드 올라옴 시 가려짐
  ├── 버튼 → 터치 영역 (44px 최소)
  └── 드롭다운/셀렉트 → 모바일 네이티브 vs 커스텀

[데이터 표시 경로]
  ├── 목록/카드 → 모바일에서 정보 밀도
  ├── 차트/그래프 → 모바일 축소 가독성
  └── 긴 텍스트 → 말줄임/접기 처리

[인터랙션 경로]
  ├── 터치 제스처 → 스와이프, 롱프레스
  ├── 스크롤 → 무한 스크롤, 풀투리프레시
  └── 입력 → 키보드 타입, 자동완성
```

### 1-3. 모바일 시나리오 매트릭스

```

Y축 (디바이스):  iPhone SE(375) | iPhone 13(390) | Galaxy(360) | iPad mini(768)
Z축 (상태):     빈 데이터 | 데이터 많음 | 키보드 올림 | 가로 모드 | 느린 네트워크

시나리오 도출 (최소 20개):
  MOB-01: 헤더 - iPhone SE에서 제목 잘림

  MOB-04: 폼 - Galaxy에서 버튼 터치 영역
  MOB-05: 카드 - 빈 데이터일 때 레이아웃
  ...
```

## Phase 2: DO (4개 에이전트 팀 동시 출동)

### 팀 1: 모바일 QA 엔지니어 (Playwright 에이전트) — 체크리스트 LAY/TCH/RD 담당
375x812 (iPhone) 해상도로 **전 페이지 + 모달 스크린샷** 촬영.
시나리오 매트릭스의 각 시나리오 실행 + 스크린샷 캡처.

### 팀 2: 프론트엔드 아키텍트 (code-analyzer 에이전트) — 체크리스트 RES 담당
영향도 맵의 "레이아웃 경로 + 컴포넌트 경로"를 따라 코드 분석.

### 팀 3: UX 디자이너 (frontend-architect 에이전트) — 체크리스트 TCH/RD 담당
영향도 맵의 "인터랙션 경로"를 따라 UX 평가.

### 팀 4: 성능 엔지니어 (general-purpose 에이전트) — 체크리스트 PF 담당
모바일 성능 측정 + 최적화 기회 식별.

## Phase 3: CHECK (점수화 + 교차 검증)

### 3-1. 체크리스트 결과 취합
```
━━━ 모바일 체크리스트 결과 ━━━
[레이아웃] N/N PASS (XX%)
[터치/인터랙션] N/N PASS (XX%)
[가독성] N/N PASS (XX%)
[성능] N/N PASS (XX%)
[반응형 코드] N/N PASS (XX%)
━━━━━━━━━━━━━━━━━━━━━━
전체: N/N PASS (XX%)
```

### 3-2. 교차 검증
- 팀1(스크린샷 정상) vs 팀2(코드에 고정 width) → **다른 기기에서 깨질 수 있음**
- 팀2(코드 정상) vs 팀3(UX 불편) → **기술적으로는 OK이지만 사용성 문제**
- 팀1(화면 정상) vs 팀4(느림) → **보이지만 로딩이 느린 문제**

### 3-3. 점수화 (100점)
| 감점 기준 | 점수 |
|----------|------|
| Critical (깨짐/기능불가) 1건당 | -15점 |
| High (불편/가독성) 1건당 | -8점 |
| Medium (개선) 1건당 | -3점 |
| Low (제안) 1건당 | -1점 |

### 3-4. 판정
- **90점 이상**: Phase 5(보고서)
- **90점 미만**: Phase 4(Act)

---

## Phase 4: ACT (수정 + 재촬영)

**최대 3회 반복.**
1. Critical/High 수정 → Playwright 재촬영
2. **수정 후 영향도 맵 연결 컴포넌트도 재확인**
3. 빌드 확인 → 커밋 → 재점검

## 핵심 규칙

1. **영향도 맵 필수**: 레이아웃/컴포넌트/데이터표시/인터랙션 4개 경로 추적
2. **시나리오 매트릭스 필수**: 화면 요소 × 디바이스 × 상태 조합 최소 20개
3. **체크리스트 필수**: 5개 카테고리 체크 항목 + 결과 기록
4. **교차 검증 필수**: 4팀 결과 교차 대조
5. **375x812 해상도** 고정 (iPhone 기준)
6. **PDCA 자동 반복**: 90점 미만 시 수정 → 재점검 최대 3회
7. **스크린샷 필수 + Before/After 비교**
