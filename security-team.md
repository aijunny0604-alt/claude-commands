# 보안 에이전트 팀 - PDCA 보안 점검

당신은 보안 에이전트 팀 리더입니다. 3개 전문 에이전트 팀을 **동시에** 출동시켜 **시나리오 매트릭스 + 영향도 맵 + 체크리스트** 기반 PDCA 사이클로 보안 점검 → 수정 → 재점검을 **90점 이상**까지 자동 반복합니다.

인자: $ARGUMENTS (점검 범위: "전체" 또는 특정 영역)

---

## PDCA 사이클 개요

```
Plan(점검계획+영향도맵) → Do(3팀 동시 점검) → Check(점수+교차검증) → Act(수정) → 재점검...
    ↑                                                                    |
    └──────────── 90점 미만이면 반복 (최대 3회) ────────────────────────┘
```

---

## Phase 1: PLAN (점검 계획 + 영향도 맵)

### 1-1. 프로젝트 기술 스택 파악
- 프레임워크, ORM, 인증 방식, 미들웨어 확인
- package.json 의존성 확인

### 1-2. 보안 영향도 맵 생성

```
📦 보안 영향도 맵 (Security Impact Map)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
[인증 경로]
  ├── 로그인 API → 세션/쿠키 → 인증 미들웨어 → 보호된 API
  ├── 인증 없이 접근 가능한 공개 API 목록
  └── 세션 만료 → 재인증 → 상태 유지

[데이터 흐름 경로]
  ├── 사용자 입력 → API → 검증(Zod?) → DB 저장
  ├── DB 조회 → API 응답 → 프론트 표시 (민감 데이터 노출?)
  └── 파일 업로드 → 저장 → 접근 제어

[외부 통신 경로]
  ├── 환경변수 → API 키 → 외부 서비스 호출
  └── CORS 설정 → 허용 도메인

[에러 경로]
  └── 에러 발생 → 에러 핸들링 → 응답 (내부 정보 노출?)
```

### 1-3. 보안 시나리오 매트릭스

```
📊 보안 시나리오 매트릭스
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
X축 (공격 벡터): Injection | XSS | CSRF | 인증 우회 | 권한 상승 | 정보 노출
Y축 (진입점):   API | 폼 입력 | URL 파라미터 | 쿠키 | 헤더 | 파일 업로드
Z축 (인증 상태): 비인증 | 일반 사용자 | 관리자 | 만료된 세션

도출 시나리오 (최소 25개):
  SEC-01: SQL Injection via API body
  SEC-02: XSS via 사용자 입력 필드
  SEC-03: CSRF 토큰 없이 POST 요청
  SEC-04: 비인증 상태로 보호된 API 접근
  SEC-05: 다른 사용자 데이터 접근 (IDOR)
  SEC-06: 에러 응답에서 스택 트레이스 노출
  SEC-07: 환경변수/API 키 하드코딩
  SEC-08: 쿠키에 httpOnly/secure 미설정
  ...
```

### 1-4. 보안 체크리스트 생성

```
━━━ 보안 마스터 체크리스트 ━━━
[인증/세션] (팀1)
  ☐ AUTH-01: 비밀번호 하드코딩 여부
  ☐ AUTH-02: 세션 토큰 보안 (httpOnly, secure, sameSite)
  ☐ AUTH-03: 세션 만료 처리
  ☐ AUTH-04: 비인증 접근 차단 (모든 보호 API)

[입력 검증] (팀1+팀2)
  ☐ INPUT-01: 모든 API에 Zod/validation 적용 여부
  ☐ INPUT-02: SQL/NoSQL Injection 방어
  ☐ INPUT-03: XSS 방어 (입력 sanitize)
  ☐ INPUT-04: Command Injection 방어
  ☐ INPUT-05: 파일 업로드 검증

[API 보호] (팀1)
  ☐ API-01: CORS 설정 적절성
  ☐ API-02: 레이트리밋 적용
  ☐ API-03: 보안 헤더 (HSTS, CSP, X-Frame-Options)
  ☐ API-04: 에러 응답에 내부 정보 미노출

[민감 데이터] (팀2)
  ☐ DATA-01: 환경변수 분리 (.env)
  ☐ DATA-02: API 키/시크릿 하드코딩 없음
  ☐ DATA-03: 민감 필드 API 응답에서 제외
  ☐ DATA-04: 로그에 민감 데이터 미포함

[위험 패턴] (팀2)
  ☐ RISK-01: eval() 사용 없음
  ☐ RISK-02: dangerouslySetInnerHTML 사용 없음
  ☐ RISK-03: SSRF 위험 없음
  ☐ RISK-04: 프로토타입 오염 위험 없음

[의존성] (팀3)
  ☐ DEP-01: npm audit 취약점 0건
  ☐ DEP-02: 알려진 CVE 없음
  ☐ DEP-03: 최신 보안 패치 적용

각 항목별 영향도 맵 연결 경로 명시
```

### 1-5. 계획서 저장
파일: `docs/03-analysis/security-plan-{날짜}.md`

## Phase 2: DO (3개 에이전트 팀 동시 출동)

### 팀 1: 보안 아키텍트 (security-architect 에이전트)

**체크리스트 AUTH-xx, INPUT-xx, API-xx 담당**

영향도 맵의 "인증 경로 + 데이터 흐름 경로"를 따라 OWASP Top 10 스캔:
- Injection, Broken Authentication, Sensitive Data Exposure
- Broken Access Control, Security Misconfiguration, XSS, CSRF
- 레이트리밋, 쿠키 보안, 입력 검증 커버리지

**시나리오 매트릭스의 각 공격 벡터를 실제 코드에서 검증**
심각도 등급 (Critical/High/Medium/Low) + 파일:줄번호 + 체크리스트 결과

### 팀 2: 코드 분석반 (code-analyzer 에이전트)

**체크리스트 DATA-xx, RISK-xx 담당**

영향도 맵의 "데이터 흐름 + 에러 경로"를 따라 코드 리뷰:
- 하드코딩 시크릿, eval, dangerouslySetInnerHTML
- 에러 메시지 정보 노출, SSRF 위험
- DELETE/PUT 권한 검증, npm audit
- 입력 검증 없는 API 목록화

**각 위험 패턴에 대해 영향도 맵의 데이터 흐름을 추적하여 실제 악용 가능성 판단**

### 팀 3: 트렌드 조사반 (general-purpose 에이전트, 웹 검색)

**체크리스트 DEP-xx 담당**

현재 사용 중인 패키지의 CVE 조사:
- 프레임워크 최신 CVE, ORM 보안 이슈
- 기타 의존성 보안 권고
- 최신 웹앱 보안 트렌드

### Playwright 보안 UI 테스트 (전 팀 공통 보조)

`playwright-cli`로 **실제 브라우저에서 보안 시나리오** 검증:
- **비인증 접근**: 로그인 없이 보호된 URL 직접 접속 → 리디렉션 확인
- **세션 만료**: 쿠키 삭제 후 페이지 새로고침 → 재인증 요구 확인
- **XSS 입력**: 폼에 `<script>alert(1)</script>` 입력 → 실행 안 됨 확인
- **URL 조작**: 존재하지 않는 경로 접속 → 404 에러 페이지 (내부 정보 미노출)
- **로그인 시도**: 잘못된 비밀번호 → 에러 메시지 (비밀번호 힌트 미포함)

## Phase 3: CHECK (점수화 + 교차 검증)

### 3-1. 체크리스트 결과 취합
```
━━━ 보안 체크리스트 결과 ━━━
[인증/세션] 팀1: N/N PASS (XX%)
[입력 검증] 팀1+2: N/N PASS (XX%)
[API 보호] 팀1: N/N PASS (XX%)
[민감 데이터] 팀2: N/N PASS (XX%)
[위험 패턴] 팀2: N/N PASS (XX%)
[의존성] 팀3: N/N PASS (XX%)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
전체: N/N PASS (XX%)
```

### 3-2. 교차 검증
- 팀1(아키텍처)에서 "인증 정상"인데 팀2(코드)에서 "하드코딩 비밀번호" → **인증 우회 가능**
- 팀2(코드)에서 "입력 검증 있음"인데 팀1(아키텍처)에서 "일부 API 미적용" → **부분 보호**
- 팀3(CVE)에서 "취약점 발견"인데 팀2(코드)에서 "해당 기능 미사용" → **실제 영향 없음**

### 3-3. 보안 점수 산출 (100점 만점)
| 감점 기준 | 점수 |
|----------|------|
| Critical 1건당 | -15점 |
| High 1건당 | -8점 |
| Medium 1건당 | -3점 |
| Low 1건당 | -1점 |

### 3-4. 영향도 맵 기반 위험도 평가
각 취약점에 대해 영향도 맵의 경로를 따라 **실제 악용 시나리오** 평가:
```
❌ AUTH-01: 비밀번호 하드코딩
  → 영향도 추적: 로그인 API → 전체 세션 → 모든 보호 API
  → 실제 위험: 소스코드 접근 시 전체 시스템 접근 가능
  → 심각도: Critical (영향 범위: 전체)
```

### 3-5. 판정
- **90점 이상 + Critical 0건**: Phase 5(보고서)
- **90점 미만 또는 Critical 있음**: Phase 4(Act)

---

## Phase 4: ACT (자동 수정 + 재점검)

**최대 3회 반복. 사용자 확인 후 진행.**

1. Critical → 즉시 수정
2. High → 즉시 수정
3. Medium → 사용자 확인 후 수정
4. **수정 후 영향도 맵의 연결 경로 재검증**
5. 빌드 확인 + 커밋
6. 수정된 항목만 재스캔 → 점수 재산출

## 핵심 규칙

1. **영향도 맵 필수**: 인증/데이터/외부통신/에러 4개 경로 반드시 추적
2. **시나리오 매트릭스 필수**: 공격 벡터 × 진입점 × 인증 상태 조합 최소 25개

4. **교차 검증 필수**: 3팀 결과 교차 대조
5. **PDCA 자동 반복**: 90점 미만 시 수정 → 재점검 최대 3회
6. **Critical은 반드시 0건**: Critical 남으면 PASS 불가
7. **빌드 테스트** 필수

## Phase X+1: Codex 대조 검증 게이트 (자동 — 선택 사항)

> 이 게이트는 v2.1부터 모든 J-AGENTS 에이전트에 공통 적용됩니다. **Codex CLI가 없으면 자동 스킵** 되므로 클로드 단독 환경에서도 그대로 동작합니다.

### 게이트 1단계: Codex 가용성 자동 체크

```bash
codex --version >/dev/null 2>&1 && echo "codex-ready" || echo "codex-missing"
```

- `codex-ready` → 게이트 2단계 진행
- `codex-missing` → 이 게이트를 통째로 스킵하고 다음 Phase(보통 REPORT)로 바로 진행. 사용자에게 "Codex 비활성 → 클로드 단독 결과만 보고합니다" 한 줄만 안내.

### 게이트 2단계: Codex에 2차 의견 요청 (대조 검증)

이번 사이클의 **핵심 산출물 3가지** (체크리스트 결과 / 핫스팟 Top 5 / 권고안)를 그대로 묶어 Codex에 보내고 같은 시각으로 재검토 요청.

- 호출: `Skill codex:rescue` 사용. 프롬프트 예시:
  > "방금 클로드가 [에이전트명] 에이전트로 도출한 아래 결과를 같은 기준으로 한 번 더 검토해줘.
  > - 누락된 항목 있는지
  > - 우선순위가 잘못 매겨진 항목 있는지
  > - 더 안전하거나 더 효율적인 대안이 있는지
  > 짧게 답변 (불릿 5개 이내)."

- Codex 응답이 1분 안에 안 오면 → 타임아웃 처리, "Codex 응답 없음 — 클로드 단독 권고로 진행" 명시 후 다음 Phase로.

### 게이트 3단계: 두 의견 합치표 (필수)

| 항목 | 클로드 1차 결과 | Codex 2차 의견 | 합치 / 충돌 | 최종 권고 |
|---|---|---|---|---|
| 핫스팟 #1 | ... | ... | ✅ 합치 | 클로드 권고 그대로 진행 |
| 핫스팟 #2 | ... | ... | ⚠️ 충돌 | 사용자에게 둘 중 선택 요청 |
| 누락 발견 | (없음) | Codex가 추가 항목 1개 제시 | ➕ 보강 | 보강안 함께 보고 |

- **합치 항목** : 자동으로 다음 Phase 진행 권고
- **충돌 항목** : `AskUserQuestion` 도구로 사용자에게 두 안 중 선택 요청
- **Codex 보강 항목** : 클로드가 놓친 부분 → 사용자에게 보강안으로 추가 보고

### 게이트 4단계: 최종 보고에 반영

다음 Phase(REPORT) 보고서에 "Codex 대조 검증 결과" 섹션을 1단락 추가:
- Codex 비활성: "Codex 단계 스킵 — 클로드 단독 분석"
- Codex 활성 + 전체 합치: "Codex 검증 통과 — 추가 권고 없음"
- Codex 활성 + 충돌/보강 있음: 합치표 그대로 보고서에 포함

### 핵심 원칙

1. **Codex 부재 = 무조건 스킵**. 패치 이후에도 클로드 단독 동작 100% 보장.
2. **Codex 의견은 권고일 뿐 자동 적용 금지**. 충돌 시 항상 사용자 결정 우선.
3. **타임아웃 1분**. Codex가 멈춰도 메인 에이전트가 막히지 않도록.
4. **대조 결과는 반드시 보고서에 기록**. 추적 가능성 유지.
