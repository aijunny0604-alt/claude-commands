# 빠른 버그 수정 에이전트 - PDCA 경량 디버깅

사용자가 보고한 버그를 빠르게 찾아 수정합니다. 원인 분석 → 수정 → 빌드 확인 → 테스트까지 자동 진행.

인자: $ARGUMENTS (버그 설명 또는 에러 메시지)

---

## PDCA 경량 사이클

```
Plan(원인 추적) → Do(수정) → Check(빌드+테스트) → Act(재수정 or 완료)
```

## Phase 1: PLAN (원인 추적)

### 1-1. 에러/버그 분석
$ARGUMENTS에서 키워드 추출 → 관련 파일 검색:
```bash
grep -rn "키워드" src/
```

### 1-2. 연관 파일 파악
- 에러가 발생하는 페이지/API
- 해당 코드가 import하는 모듈
- 관련 타입 정의
- DB 모델 (필요 시)

### 1-3. 원인 진단
- 타입 에러 → 타입 정의 확인
- API 에러 → 요청/응답 흐름 추적
- UI 버그 → 상태 관리 + 렌더링 로직 확인
- 연동 버그 → 트랜잭션 + 외부 API 확인

## Phase 2: DO (수정)

### 2-1. 최소 범위 수정
- 버그 원인만 정확히 수정
- 불필요한 리팩토링 금지
- 기존 동작에 영향 없는지 확인

### 2-2. 수정 내역 기록
```
파일: xxx.tsx (줄 N)
원인: [원인]
수정: [수정 내용]
```

## Phase 3: CHECK (검증)

### 3-1. 빌드 확인
```bash
npx next build
```

### 3-2. 관련 API 테스트
수정된 기능의 API를 curl로 직접 호출하여 정상 동작 확인.

### 3-3. 연관 기능 사이드이펙트 확인
수정이 다른 기능에 영향을 미치지 않는지 확인.

## Phase 4: ACT (판정)

- **빌드 성공 + 테스트 통과**: 커밋 안내
- **빌드 실패 또는 테스트 실패**: Phase 2로 돌아가 재수정 (최대 3회)

## 보고 형식

```
## 버그 수정 완료

### 원인
[원인 설명] — 파일:줄

### 수정
[수정 내용] — 파일:줄

### 검증
- 빌드: ✅
- API 테스트: ✅
- 사이드이펙트: ✅

### 커밋 준비 완료
```

## 규칙
1. **최소 수정**: 버그만 고침, 리팩토링 금지
2. **빌드 필수**: 수정 후 빌드 통과 확인
3. **테스트 필수**: 관련 API 최소 1개 호출 확인
4. **3회 실패 시**: 사용자에게 수동 확인 요청
