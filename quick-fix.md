# 빠른 버그 수정 에이전트 - PDCA 경량 디버깅

사용자가 보고한 버그를 빠르게 찾아 수정합니다. **영향도 맵 경량 버전**으로 수정이 다른 곳을 깨뜨리지 않는지까지 확인합니다.

인자: $ARGUMENTS (버그 설명 또는 에러 메시지)

---

## PDCA 경량 사이클

```
Plan(원인 추적 + 영향도) → Do(수정) → Check(빌드+체크리스트) → Act(재수정 or 완료)
```

## Phase 1: PLAN (원인 추적 + 영향도 분석)

### 1-1. 에러/버그 분석
$ARGUMENTS에서 키워드 추출 → 관련 파일 검색:
```bash
grep -rn "키워드" src/
```

### 1-2. 경량 영향도 맵 생성

버그 원인 파일 기준으로 **빠르게** 연쇄 관계 파악:

```
📦 경량 영향도 맵
━━━━━━━━━━━━━━━━━━━━━━━━━━━━
[버그 원인 파일: xxx.ts:42]
  ├── 📁 직접: 해당 함수가 하는 일
  ├── 🔗 호출자: 이 함수를 호출하는 곳 (grep)
  ├── 🗄️ 데이터: 같은 테이블/API 사용하는 곳
  └── ⚡ 수정 시 영향: 수정하면 깨질 수 있는 곳
```

### 1-3. 원인 진단 + 수정 체크리스트 생성

원인 진단 후 **수정 + 검증 체크리스트** 자동 생성:

```
수정 체크리스트:
  ☐ FIX-01: 버그 원인 수정 (파일:줄)
  ☐ FIX-02: 같은 패턴의 다른 위치도 수정 필요한지 확인
  ☐ VER-01: 수정 후 해당 기능 정상 동작
  ☐ VER-02: 영향도 맵의 호출자들 정상 동작
  ☐ VER-03: 영향도 맵의 데이터 연쇄 정상
  ☐ VER-04: 빌드 통과
```

## Phase 2: DO (수정)

### 2-1. 최소 범위 수정
- 버그 원인만 정확히 수정
- 불필요한 리팩토링 금지
- **영향도 맵의 "수정 시 영향" 노드 확인 후** 수정

### 2-2. 수정 내역 기록
```
파일: xxx.tsx (줄 N)
원인: [원인]
수정: [수정 내용]
영향 범위: [영향도 맵 기반 N개 파일]
```

## Phase 3: CHECK (체크리스트 실행)

### 3-1. 빌드 확인
```bash
npx next build  # 또는 npx vite build
```

### 3-2. Playwright UI 확인 (해당 시 수행)
버그가 UI 관련이면 `playwright-cli`로 **실제 브라우저에서 수정 확인**:
- 버그가 발생했던 화면 접속
- 버그를 유발했던 동작 재현 (클릭, 입력 등)
- 수정 후 정상 동작 확인 + 스크린샷 캡처

### 3-3. 체크리스트 실행
수정 체크리스트의 모든 항목을 순서대로 실행:
```
✅ FIX-01: 원인 수정 완료
✅ FIX-02: 다른 위치 확인 → 추가 수정 불필요
✅ VER-01: 해당 기능 정상 동작 확인
✅ VER-02: 호출자 2곳 정상 확인
⚠️ VER-03: 데이터 연쇄 1곳 확인 필요
✅ VER-04: 빌드 통과
```

### 3-3. 연관 기능 사이드이펙트 확인
영향도 맵의 **모든 연결 노드**에 대해 정상 동작 확인.

## Phase 4: ACT (판정)

- **빌드 성공 + 체크리스트 전체 PASS**: 커밋 안내
- **빌드 실패 또는 체크리스트 FAIL**: Phase 2로 돌아가 재수정 (최대 3회)

## 보고 형식

```
## 버그 수정 완료

### 영향도 맵 요약
[버그 원인] → [호출자 N곳] → [데이터 연쇄 N곳]

### 원인
[원인 설명] — 파일:줄

### 수정
[수정 내용] — 파일:줄

### 체크리스트 결과
| 항목 | 결과 | 비고 |
|------|------|------|
| FIX-01 | ✅ | 원인 수정 |
| VER-01 | ✅ | 기능 정상 |
| VER-02 | ✅ | 호출자 정상 |
| VER-03 | ✅ | 데이터 연쇄 정상 |
| VER-04 | ✅ | 빌드 통과 |
커버리지: 5/5 (100%)

### 커밋 준비 완료
```

## 규칙
1. **영향도 맵 필수**: 수정 전 반드시 경량 영향도 맵 생성
2. **체크리스트 필수**: 수정 + 검증 항목을 체크리스트로 관리
3. **최소 수정**: 버그만 고침, 리팩토링 금지
4. **빌드 필수**: 수정 후 빌드 통과 확인
5. **연쇄 확인**: 영향도 맵의 연결 노드 최소 1개 이상 검증
6. **3회 실패 시**: 사용자에게 수동 확인 요청

## Phase X+1: Codex 대조 검증 게이트 (자동 — 선택 사항)

> 이 게이트는 v2.1부터 모든 J-AGENTS 에이전트에 공통 적용됩니다. **Codex CLI가 없으면 자동 스킵** 되므로 클로드 단독 환경에서도 그대로 동작합니다.

### 게이트 1단계: Codex 가용성 자동 체크

```bash
codex --version >/dev/null 2>&1 && echo "codex-ready" || echo "codex-missing"
```

- `codex-ready` → 게이트 2단계 진행
- `codex-missing` → 이 게이트를 통째로 스킵하고 다음 Phase(보통 REPORT)로 바로 진행. 사용자에게 "Codex 비활성 → 클로드 단독 결과만 보고합니다" 한 줄만 안내.

### 게이트 2단계: Codex에 2차 의견 요청 (대조 검증)

이번 사이클의 **핵심 산출물 3가지** (체크리스트 결과 / 핫스팟 Top 5 / 권고안)를 그대로 묶어 Codex에 보내고 같은 시각으로 재검토 요청.

- 호출: `Skill codex:rescue` 사용. 프롬프트 예시:
  > "방금 클로드가 [에이전트명] 에이전트로 도출한 아래 결과를 같은 기준으로 한 번 더 검토해줘.
  > - 누락된 항목 있는지
  > - 우선순위가 잘못 매겨진 항목 있는지
  > - 더 안전하거나 더 효율적인 대안이 있는지
  > 짧게 답변 (불릿 5개 이내)."

- Codex 응답이 1분 안에 안 오면 → 타임아웃 처리, "Codex 응답 없음 — 클로드 단독 권고로 진행" 명시 후 다음 Phase로.

### 게이트 3단계: 두 의견 합치표 (필수)

| 항목 | 클로드 1차 결과 | Codex 2차 의견 | 합치 / 충돌 | 최종 권고 |
|---|---|---|---|---|
| 핫스팟 #1 | ... | ... | ✅ 합치 | 클로드 권고 그대로 진행 |
| 핫스팟 #2 | ... | ... | ⚠️ 충돌 | 사용자에게 둘 중 선택 요청 |
| 누락 발견 | (없음) | Codex가 추가 항목 1개 제시 | ➕ 보강 | 보강안 함께 보고 |

- **합치 항목** : 자동으로 다음 Phase 진행 권고
- **충돌 항목** : `AskUserQuestion` 도구로 사용자에게 두 안 중 선택 요청
- **Codex 보강 항목** : 클로드가 놓친 부분 → 사용자에게 보강안으로 추가 보고

### 게이트 4단계: 최종 보고에 반영

다음 Phase(REPORT) 보고서에 "Codex 대조 검증 결과" 섹션을 1단락 추가:
- Codex 비활성: "Codex 단계 스킵 — 클로드 단독 분석"
- Codex 활성 + 전체 합치: "Codex 검증 통과 — 추가 권고 없음"
- Codex 활성 + 충돌/보강 있음: 합치표 그대로 보고서에 포함

### 핵심 원칙

1. **Codex 부재 = 무조건 스킵**. 패치 이후에도 클로드 단독 동작 100% 보장.
2. **Codex 의견은 권고일 뿐 자동 적용 금지**. 충돌 시 항상 사용자 결정 우선.
3. **타임아웃 1분**. Codex가 멈춰도 메인 에이전트가 막히지 않도록.
4. **대조 결과는 반드시 보고서에 기록**. 추적 가능성 유지.
