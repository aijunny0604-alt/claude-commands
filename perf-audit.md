# 성능 점검 에이전트 팀 - PDCA 성능 최적화

당신은 성능 최적화 전문 에이전트 팀 리더입니다. 3개 전문 에이전트 팀을 **동시에** 출동시켜 **시나리오 매트릭스 + 영향도 맵 + 체크리스트** 기반 PDCA 사이클로 성능 점검 → 최적화 → 재점검을 **90점 이상**까지 자동 반복합니다.

인자: $ARGUMENTS (점검 대상: "전체" 또는 특정 페이지/API)

---

## Phase 1: PLAN

### 1-1. 성능 영향도 맵

```
📦 성능 영향도 맵
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
[렌더링 경로 (Critical Rendering Path)]
  ├── 초기 로드: HTML → CSS → JS → First Paint → LCP
  ├── 번들: 각 라우트별 JS 크기
  └── 이미지/폰트: 블로킹 리소스

[API 경로]
  ├── 각 API 엔드포인트 → Prisma 쿼리 → DB
  ├── 직렬 호출 (순차) vs 병렬 호출 (Promise.all)
  └── 캐싱 기회: 자주 호출 + 변경 적은 데이터

[리렌더링 경로]
  ├── 상태 변경 → 리렌더 트리거 → 자식 리렌더
  ├── 메모이제이션 누락: useMemo, useCallback 미적용
  └── 불필요 리렌더: 부모 변경에 의한 전체 리렌더

[데이터 흐름 병목]
  ├── N+1 쿼리: 루프 내 DB 호출
  ├── 과다 데이터: 전체 조회 후 일부만 사용
  └── 무거운 연산: 클라이언트 사이드 정렬/필터
```

### 1-2. 성능 시나리오 매트릭스

```
X축 (측정 대상): 페이지 로드 | API 응답 | 번들 크기 | 리렌더링
Y축 (페이지/API): 각 라우트, 각 API
Z축 (부하 상태): 빈 데이터 | 정상 | 대량 데이터 | 동시 요청

도출 (최소 20개):
  PF-01: 대시보드 로드 - 대량 데이터(1000건+) 시 LCP
  PF-02: /api/reservations GET - 데이터 100건 응답 시간
  PF-03: 예약 목록 - 스크롤 시 리렌더링 횟수
  PF-04: 번들 - 모달 컴포넌트 lazy load 가능?
  ...
```

### 1-3. 성능 체크리스트

```
━━━ 성능 마스터 체크리스트 ━━━
[페이지 로드] (팀1)
  ☐ PL-01: LCP 2.5초 이하
  ☐ PL-02: FCP 적정
  ☐ PL-03: CLS 0.1 이하
  ☐ PL-04: DOM 요소 1500개 이하
  ☐ PL-05: 이미지 최적화 (next/image)
  ☐ PL-06: 폰트 블로킹 없음

[번들] (팀2)
  ☐ BD-01: First Load JS 100KB 이하 (라우트별)
  ☐ BD-02: 불필요 import 없음
  ☐ BD-03: tree-shaking 정상
  ☐ BD-04: 동적 import 활용 (모달, 차트)
  ☐ BD-05: 미사용 에셋 없음

[API 응답] (팀3)
  ☐ AP-01: 각 API 200ms 이하
  ☐ AP-02: N+1 쿼리 없음
  ☐ AP-03: select 사용 (필요 필드만)
  ☐ AP-04: Promise.all 활용 (병렬 가능한 곳)
  ☐ AP-05: 캐싱 적용 가능한 곳 식별

[리렌더링] (팀2)
  ☐ RR-01: 불필요 리렌더링 없음
  ☐ RR-02: useMemo/useCallback 적절 사용
  ☐ RR-03: 큰 목록 가상화 (필요 시)
```

## Phase 2: DO (3팀 동시 출동)

### 팀 1: Lighthouse + Core Web Vitals — PL 체크리스트 담당
Playwright로 각 페이지 성능 측정. 시나리오 매트릭스의 부하 상태별 측정.

### 팀 2: 번들 + 프론트엔드 — BD/RR 체크리스트 담당
영향도 맵의 "렌더링 경로 + 리렌더링 경로" 따라 코드 분석.

### 팀 3: API + DB 응답 시간 — AP 체크리스트 담당
영향도 맵의 "API 경로 + 데이터 흐름 병목" 따라 curl 측정.

## Phase 3: CHECK (점수화 + 교차 검증)

### 3-1. 체크리스트 결과 + 커버리지
### 3-2. 교차 검증
- 팀1(페이지 느림) + 팀3(API 느림) → **API가 병목 원인**
- 팀1(페이지 느림) + 팀2(번들 큼) → **번들이 병목 원인**
- 팀2(리렌더 과다) + 팀3(API 정상) → **프론트엔드 최적화 필요**

### 3-3. 영향도 맵 기반 병목 경로 시각화
```

  → 원인 추적: /api/dashboard (1.5초) → findMany 전체 스캔 → 인덱스 없음
```

## 핵심 규칙

1. **영향도 맵 필수**: 렌더링/API/리렌더/데이터 4개 경로 추적
2. **시나리오 매트릭스 필수**: 측정 대상 × 페이지 × 부하 상태 조합
3. **체크리스트 필수**: 18개 항목 전부 실행 + 결과 기록
4. **교차 검증 필수**: 페이지+번들+API 결과 교차 분석으로 병목 특정
5. **측정 먼저**: 추측 금지, 반드시 측정 후 최적화
6. **PDCA 자동 반복**: 90점 미만 시 최적화 → 재측정 최대 3회
