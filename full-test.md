# 대규모 통합 테스트 에이전트팀 - 로컬 + 프로덕션

당신은 QA 테스트 총괄 리더입니다. 전문 에이전트 **4~5개 팀을 동시에** 투입하여 **시나리오 매트릭스 + 영향도 맵 + 체크리스트** 기반으로 전체 기능을 로컬과 프로덕션 양쪽에서 꼼꼼하게 검증합니다.

인자: $ARGUMENTS (테스트 대상 기능 또는 "전체")

### 0-2. 영향도 맵 생성 (Impact Map)

변경된 파일 기준으로 **전체 연쇄 관계** 추적. 실제 코드를 읽어서 작성:

```
📦 영향도 맵 (Impact Map)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
[변경 파일: xxx.ts]
  ├── 📁 직접 영향 (이 파일 내부)
  │   ├── 함수: functionA(), functionB()
  │   └── 상태: stateX, stateY
  ├── 🔗 1차 연쇄 (이 파일을 import하는 곳)
  │   ├── ComponentA.tsx (props: onXxx, dataYyy)
  │   ├── ComponentB.tsx (hooks: useXxx)
  │   └── page.tsx (라우트 핸들러)
  ├── 🗄️ 데이터 연쇄 (같은 DB 테이블/API 공유)
  │   ├── /api/reservations → Reservation 테이블
  │   ├── /api/customers → Customer 테이블 (관계: 1:N)
  │   └── /api/dashboard → 통계 집계 (읽기 전용)
  ├── ⚡ 이벤트 연쇄 (콜백/이벤트 전파)
  │   ├── onSave → refreshList → revalidate
  │   ├── onDelete → confirmModal → refreshList
  │   └── onChange → debounce → search → filterResults
  └── 🔄 상태 연쇄 (공유 상태/컨텍스트)
      ├── AuthContext → 전 페이지
      ├── filterState → 목록 + 검색 + 정렬
      └── modalState → 모달 계층 관리
```

### 0-3. 시나리오 매트릭스 자동 도출

영향도 맵의 각 노드에 대해 **변수 조합으로 시나리오 자동 생성**:

```
📊 시나리오 매트릭스
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
축 정의:
  X축 (데이터 상태): 정상 | 빈 값 | 경계값 | 잘못된 타입 | null/undefined
  Y축 (CRUD 동작):  Create | Read | Update | Delete | Complete | Cancel
  Z축 (환경 조건):  첫 로드 | 기존 데이터 | 동시 작업 | 연속 작업 | 에러 후 재시도

자동 생성 규칙:
  - 영향도 맵의 각 "직접 영향" 노드 × X축 = 기본 시나리오
  - 영향도 맵의 각 "데이터 연쇄" 노드 × Y축 = 연동 시나리오
  - 영향도 맵의 각 "이벤트 연쇄" 노드 × Z축 = 연쇄 시나리오

최소 시나리오 수: 30개 이상 (기능당 최소 5개)
```

### 0-4. 마스터 체크리스트 생성

시나리오 매트릭스에서 **실행 가능한 체크리스트**로 변환:

```
━━━ 마스터 체크리스트 ━━━
[데이터 무결성] (팀1 담당)
  ☐ DI-01: 타입 안전성 - optional chaining 누락 확인
  ☐ DI-02: null/undefined 미처리 확인
  ☐ DI-03: API 요청/응답 스키마 일치 확인
  ☐ DI-04: 연동 로직 트랜잭션 정합성

[API 기능] (팀2 담당)
  ☐ AF-01: POST 정상 입력 → 201 + 데이터 반환
  ☐ AF-02: POST 필수값 누락 → 400 + 에러 메시지
  ☐ AF-03: GET 목록 → 필드 완전성
  ☐ AF-04: PUT 부분 업데이트 → 변경 필드만 반영
  ☐ AF-05: DELETE → 연관 데이터 정리 + 재고 환불

[연관성+사이드이펙트] (팀3 담당)
  ☐ SE-01: 오일 재고 차감/환불 동기화
  ☐ SE-02: 부품 재고 차감/환불 동기화
  ☐ SE-03: 매출 통계 COMPLETED만 집계
  ☐ SE-04: 캘린더 이벤트 연동

[Edge Case] (팀4 담당)
  ☐ EC-01: 동일 리소스 연속 수정 (3회)
  ☐ EC-02: 빈 값 전송 ({}, null, "")
  ☐ EC-03: 음수/NaN 값
  ☐ EC-04: 존재하지 않는 ID
  ☐ EC-05: 상태 전이 제한 위반

각 항목별 기대 결과 + 검증 방법 + 영향도 맵 연결 노드 명시
```

## Phase 2: 에이전트 4팀 동시 출동 (병렬 실행)

$ARGUMENTS이 특정 기능이면 해당 기능 중심으로, "전체"이면 아래 전 항목을 테스트.

### 팀 1: 데이터 무결성 검증반 (Explore 에이전트)

**코드 분석만 수행 (API 호출 X) — 체크리스트 DI-xx 담당**

**영향도 맵의 모든 경로를 따라** 코드 분석:
- **타입 안전성**: optional chaining 누락, null/undefined 미처리, 타입 캐스팅 위험
- **상태 관리 일관성**: useState 초기화 누락, 고객/차량 변경 시 연관 상태 초기화 여부
- **API 요청/응답 일관성**: Zod 스키마 적용 여부, include/select 필드 누락, 에러 핸들링
- **연동 로직**: 예약↔오일↔부품↔캘린더↔시트 간 트랜잭션 정합성 (영향도 맵 데이터 연쇄 기반)
- **edge case**: 동시 입력, null vs undefined, 빈 문자열, falsy 값(0, ""), NaN
- **보안**: 음수 값, 서버 검증 누락, XSS, injection 가능성

**체크리스트 DI-xx 결과를 ✅/❌/⚠️로 기록** + 파일:줄번호

### 팀 2: API 기능 테스트반 (general-purpose 에이전트)

**프로덕션 서버에서 실제 API 호출 — 체크리스트 AF-xx 담당**

**시나리오 매트릭스의 X축(데이터 상태) × Y축(CRUD)** 조합으로 테스트:
- **생성(POST)**: 정상 입력, 필수값 누락, 잘못된 값, edge case
- **조회(GET)**: 목록, 상세, 필터링, 응답 필드 완전성
- **수정(PUT)**: 부분 업데이트, 전체 업데이트, 연관 데이터 변경
- **삭제(DELETE)**: 삭제 후 연관 데이터 정리, 재고 환불
- **완료(complete)**: 상태 전이, 재고 차감, 비용 기록
- **취소(CANCELLED)**: 오일/부품 환불, mileage 롤백 없음

**각 테스트에 사용한 입력값 + 기대 결과 + 실제 결과** 기록. 테스트 후 모든 데이터 원복 필수!

### 팀 3: 연관성 + 사이드이펙트 테스트반 (general-purpose 에이전트)

**프로덕션 서버에서 연관 시스템 영향도 검증 — 체크리스트 SE-xx 담당**

**영향도 맵의 데이터 연쇄 + 이벤트 연쇄** 경로를 따라 테스트:
- **오일 재고 연동**: 예약 등록→차감, 취소→환불, 삭제→환불, 이중취소 방지
- **부품 재고 연동**: 완료→차감, 취소→환불, 삭제→환불
- **매출 통계**: COMPLETED만 집계, 테스트 데이터 영향 없음
- **대시보드**: 통계 API 정상 응답, 오늘/이번주/이번달 데이터 정합성
- **캘린더 연동**: 예약 수정 시 캘린더 이벤트도 업데이트 (fire-and-forget)
- **고객 데이터**: 고객 변경 시 예약-고객 관계 정합, 차량 목록 갱신

**연쇄 경로별 테스트: A→B→C 순서대로 실행하고 각 단계에서 데이터 확인**. 테스트 후 모든 데이터 원복 필수!

### 팀 4: Edge Case + 스트레스 테스트반 (general-purpose 에이전트)

**프로덕션 서버에서 극단적 시나리오 검증 — 체크리스트 EC-xx 담당**

**시나리오 매트릭스의 경계값 + 비정상** 조합으로 테스트:
- **같은 리소스 연속 수정**: 동일 예약 3회 연속 PUT
- **같은 차량 2개 예약**: 동시 mileage 업데이트 → 마지막 값 우선
- **빈 값 전송**: `{}`, `{mileage: null}`, `{vehicleId: ""}` 등
- **음수/NaN**: `{mileage: -1}`, `{duration: -5}`, `{cost: -100}`
- **존재하지 않는 ID**: 없는 customerId, vehicleId로 PUT
- **상태 전이 제한**: COMPLETED→PENDING 불가? CANCELLED→COMPLETED 가능?
- **대량 데이터**: 예약 5개 연속 생성 → 목록 정상 표시 → 전체 삭제

테스트 후 모든 데이터 원복 필수!

## Phase 3: 결과 종합 + 교차 검증

### 3-1. 체크리스트 결과 취합

4팀의 결과를 마스터 체크리스트에 통합:
```
━━━ 체크리스트 실행 결과 ━━━
[데이터 무결성] 팀1: 12/12 PASS (100%)
  ✅ DI-01 | ✅ DI-02 | ✅ DI-03 | ...
[API 기능] 팀2: 18/20 PASS (90%)
  ✅ AF-01 | ❌ AF-02 | ✅ AF-03 | ...
[연관성] 팀3: 8/8 PASS (100%)
  ✅ SE-01 | ✅ SE-02 | ...
[Edge Case] 팀4: 6/7 PASS (86%)
  ✅ EC-01 | ❌ EC-03 | ...
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
전체: 44/47 PASS (93.6%)
```

### 3-2. 교차 검증 (Cross-Validation)

팀 간 결과를 교차 대조하여 **모순/누락 탐지**:
- 팀1(코드)에서 "정상"인데 팀2(API)에서 "실패" → 런타임 이슈
- 팀2(API)에서 "성공"인데 팀3(연관)에서 "불일치" → 사이드이펙트 미감지
- 팀3(연관)에서 "정상"인데 팀4(Edge)에서 "실패" → 경계값 처리 누락

### 3-3. FAIL 항목 영향도 역추적

FAIL 발견 시 영향도 맵을 **역방향으로 추적**:
```
❌ AF-02 FAIL
  → 원인: 입력 검증 누락 (api/xxx/route.ts:42)
  → 영향도 맵 역추적:
    ├── 같은 검증 로직 사용하는 다른 API: /api/yyy, /api/zzz
    ├── 이 API를 호출하는 UI: ComponentA, ComponentB
    └── 관련 테이블: reservations (FK constraint 위반 가능)
  → 추가 점검 필요: AF-11, SE-03 항목도 재검증
```

### 3-4. 수정 후 재테스트 (PDCA 반복)
1. FAIL 항목 원인 파악 → 코드 확인 → 즉시 수정
2. 수정 후 해당 시나리오 + **영향도 맵 연결 시나리오** 재테스트
3. 전체 빌드 재확인
4. **최대 3회 반복**

## 핵심 규칙

1. **영향도 맵 필수**: 테스트 전 반드시 영향도 맵 생성. 맵 없이 테스트 시작 금지
2. **시나리오 매트릭스 필수**: 변수 조합으로 최소 30개 시나리오 자동 도출
3. **체크리스트 필수**: 시나리오별 실행 가능한 체크 항목 생성 + 결과 기록
4. **교차 검증 필수**: 4팀 결과를 교차 대조하여 모순/누락 탐지
5. **영향도 역추적**: FAIL 시 영향도 맵 역방향으로 파급 범위 확인
