# 대규모 통합 테스트 에이전트팀 - 로컬 + 프로덕션

당신은 QA 테스트 총괄 리더입니다. 전문 에이전트 **4~5개 팀을 동시에** 투입하여 현재 프로젝트의 전체 기능을 로컬과 프로덕션 양쪽에서 꼼꼼하게 검증하세요.

인자: $ARGUMENTS (테스트 대상 기능 또는 "전체")

---

## 사전 준비

1. **변경 파일 파악**: `git diff --stat HEAD~1` 또는 `git log --oneline -3`으로 최근 변경 내역 확인
2. **빌드 테스트**: `npx next build` 실행 → 실패 시 즉시 보고 후 중단
3. **서버 준비**:
   - 로컬: `npm run dev` (포트 3000 또는 3099)
   - 프로덕션: https://bigs-motors-manager.vercel.app
4. **로그인**: 양쪽 서버에 curl로 로그인 → 쿠키 파일 생성
   ```bash
   curl -s -c /tmp/local_cookies.txt -X POST http://localhost:3000/api/auth/login -H "Content-Type: application/json" -d '{"password":"bigsmotors2026"}'
   curl -s -c /tmp/prod_cookies.txt -X POST https://bigs-motors-manager.vercel.app/api/auth/login -H "Content-Type: application/json" -d '{"password":"bigsmotors2026"}'
   ```

## 1단계: 에이전트 4팀 동시 출동 (병렬 실행)

$ARGUMENTS이 특정 기능이면 해당 기능 중심으로, "전체"이면 아래 전 항목을 테스트.

### 팀 1: 데이터 무결성 검증반 (Explore 에이전트)

**코드 분석만 수행 (API 호출 X)**

변경된 파일과 연관된 모든 코드를 추적하여 로직 정합성 검증:
- **타입 안전성**: optional chaining 누락, null/undefined 미처리, 타입 캐스팅 위험
- **상태 관리 일관성**: useState 초기화 누락, 고객/차량 변경 시 연관 상태 초기화 여부
- **API 요청/응답 일관성**: Zod 스키마 적용 여부, include/select 필드 누락, 에러 핸들링
- **연동 로직**: 예약↔오일↔부품↔캘린더↔시트 간 트랜잭션 정합성
- **edge case**: 동시 입력, null vs undefined, 빈 문자열, falsy 값(0, ""), NaN
- **보안**: 음수 값, 서버 검증 누락, XSS, injection 가능성

파일:줄번호와 함께 보고.

### 팀 2: API 기능 테스트반 (general-purpose 에이전트)

**프로덕션 서버에서 실제 API 호출로 테스트**

변경된 기능 중심으로 CRUD 전 과정 검증:
- **생성(POST)**: 정상 입력, 필수값 누락, 잘못된 값, edge case
- **조회(GET)**: 목록, 상세, 필터링, 응답 필드 완전성
- **수정(PUT)**: 부분 업데이트, 전체 업데이트, 연관 데이터 변경
- **삭제(DELETE)**: 삭제 후 연관 데이터 정리, 재고 환불
- **완료(complete)**: 상태 전이, 재고 차감, 비용 기록
- **취소(CANCELLED)**: 오일/부품 환불, mileage 롤백 없음

테스트 후 모든 데이터 원복 필수!

### 팀 3: 연관성 + 사이드이펙트 테스트반 (general-purpose 에이전트)

**프로덕션 서버에서 연관 시스템 영향도 검증**

- **오일 재고 연동**: 예약 등록→차감, 취소→환불, 삭제→환불, 이중취소 방지
- **부품 재고 연동**: 완료→차감, 취소→환불, 삭제→환불
- **매출 통계**: COMPLETED만 집계, 테스트 데이터 영향 없음
- **대시보드**: 통계 API 정상 응답, 오늘/이번주/이번달 데이터 정합성
- **캘린더 연동**: 예약 수정 시 캘린더 이벤트도 업데이트 (fire-and-forget)
- **고객 데이터**: 고객 변경 시 예약-고객 관계 정합, 차량 목록 갱신

테스트 후 모든 데이터 원복 필수!

### 팀 4: Edge Case + 스트레스 테스트반 (general-purpose 에이전트)

**프로덕션 서버에서 극단적 시나리오 검증**

- **같은 리소스 연속 수정**: 동일 예약 3회 연속 PUT
- **같은 차량 2개 예약**: 동시 mileage 업데이트 → 마지막 값 우선
- **빈 값 전송**: `{}`, `{mileage: null}`, `{vehicleId: ""}` 등
- **음수/NaN**: `{mileage: -1}`, `{duration: -5}`, `{cost: -100}`
- **존재하지 않는 ID**: 없는 customerId, vehicleId로 PUT
- **상태 전이 제한**: COMPLETED→PENDING 불가? CANCELLED→COMPLETED 가능?
- **대량 데이터**: 예약 5개 연속 생성 → 목록 정상 표시 → 전체 삭제

테스트 후 모든 데이터 원복 필수!

## 2단계: 결과 종합 + 직접 검증

4팀 결과를 받은 후 **직접** 핵심 이슈를 재검증:
1. FAIL 항목이 있으면 원인 파악 → 코드 확인 → 즉시 수정
2. 수정 후 해당 시나리오 재테스트
3. 전체 빌드 재확인

## 핵심 규칙

1. **프로덕션 데이터 보호**: 테스트 데이터는 반드시 원복. 기존 데이터 절대 수정 금지.
2. **실제 API 호출**: curl + node -e 조합으로 실제 요청/응답 검증 (mock 금지)
3. **모든 에이전트에 쿠키 파일 경로 전달**: `-b /tmp/prod_cookies.txt`
4. **JSON 파싱**: node -e 사용 (python3 없을 수 있음)
5. **FAIL 즉시 보고**: 하나라도 실패하면 원인 분석 후 수정 → 재테스트
6. **원복 검증**: 각 에이전트가 테스트 종료 시 원복 확인 보고 포함
