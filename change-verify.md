# 변경사항 정밀 검증 에이전트팀 - PDCA 멀티 시나리오 테스트

당신은 변경사항 검증 총괄 리더입니다. 코드 수정/업데이트 후 **시나리오 매트릭스 + 영향도 맵 + 체크리스트 + 교차 검증** 기반의 **5단계 검증 프로세스**를 통해 변경이 올바르게 작동하는지, 기존 기능을 깨뜨리지 않는지 철저히 검증합니다.

인자: $ARGUMENTS (변경된 기능 설명 또는 "auto"로 git diff 자동 감지)

### 0-2. 영향도 맵 생성 (Impact Map)

변경된 **모든 파일**에 대해 실제 코드를 읽고 연쇄 관계 추적:

```
📦 영향도 맵 (Impact Map)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
[변경 파일 1: xxx.ts]
  ├── 📁 직접 영향
  │   ├── 변경된 함수: functionA() (줄 42-68 수정)
  │   ├── 변경된 타입: InterfaceX (필드 추가)
  │   └── 변경된 상태: stateA (초기값 변경)
  ├── 🔗 1차 연쇄 (이 파일을 import하는 곳)
  │   ├── ComponentA.tsx → functionA() 호출
  │   ├── ComponentB.tsx → InterfaceX 타입 사용
  │   └── api/route.ts → functionA() 호출
  ├── 🔗 2차 연쇄 (1차 파일을 import하는 곳)
  │   ├── PageX.tsx → ComponentA 렌더링
  │   └── PageY.tsx → ComponentB 렌더링
  ├── 🗄️ 데이터 연쇄
  │   ├── reservations 테이블 (CRUD 영향)
  │   ├── customers 테이블 (FK 관계)
  │   └── /api/dashboard (읽기 영향)
  ├── ⚡ 이벤트 연쇄
  │   ├── onSave → refreshList → 목록 갱신
  │   └── onChange → validate → 에러 표시
  └── 🔄 상태 연쇄
      ├── filterState → 검색+정렬 영향
      └── modalState → 모달 열림/닫힘 영향

[변경 파일 2: yyy.ts]
  └── ... (동일 구조)

━━━ 교차 영향 (Cross Impact) ━━━
  파일1 ←→ 파일2: 공유 함수 functionA()를 통해 연결
  파일1 → 파일3 (미변경): reservations 테이블 공유
```

**반드시** 실제 코드의 import/export, 함수 호출, 타입 참조를 추적할 것.

### 0-3. 시나리오 매트릭스 도출

영향도 맵의 각 노드에 대해 **변수 조합으로 시나리오 자동 생성**:

```
📊 시나리오 매트릭스
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
축 정의:
  X축 (데이터 상태): 정상값 | 빈 값("", null, undefined) | 경계값(0, -1, MAX) | 잘못된 타입
  Y축 (사용자 행동): 변경된 기능 사용 | 기존 기능 사용 | 연속 작업 | 되돌리기
  Z축 (환경 조건): 첫 로드 | 데이터 존재 | 에러 후 재시도 | 새로고침 후

도출 규칙:
  - 변경된 함수 × X축 → "변경 정확성" 시나리오 (최소 8개)
  - 영향도 1차 연쇄 × Y축 → "기존 기능 보호" 시나리오 (최소 8개)
  - 영향도 데이터/이벤트 연쇄 × Z축 → "연쇄반응" 시나리오 (최소 8개)
  - X축 경계값 + 잘못된 타입 → "시나리오 폭격" (최소 10개)
  
  총 최소: 34개 시나리오
```

  ☐ CR-03: 상태 연쇄 (공유 상태) 정상

## Phase 1: 에이전트 4팀 동시 출격 (병렬)

### 팀 1: 변경 정확성 검증반 (Explore 에이전트)

**"변경이 의도대로 되었는가?" — 체크리스트 CA-xx 담당**

영향도 맵의 "직접 영향" 노드를 중심으로 코드 분석:
- **의도 일치**: 변경 목적과 실제 코드 수정이 일치하는지
- **누락 확인**: 같은 패턴이 사용되는 **모든 위치** (영향도 맵 기반) 확인
- **타입 안전성**: 변경으로 인한 타입 불일치, optional chaining 누락
- **로직 정합성**: if/else 분기, 조건문, 계산식이 올바른지
- **상태 관리**: useState/useEffect 의존성 배열, 상태 초기화 누락
- **하드코딩**: 변경하면서 하드코딩된 값이 남아있지 않은지

**체크리스트 CA-xx 결과 ✅/❌/⚠️ 기록** + 파일:줄번호

### 팀 2: 기존 기능 보호반 (general-purpose 에이전트)

**"기존에 잘 되던 것이 깨지지 않았는가?" — 체크리스트 FP-xx 담당**

영향도 맵의 "1차 연쇄 + 2차 연쇄" 경로를 따라:
- **API 정상 응답**: 변경 파일의 **모든** API 엔드포인트 호출 테스트
- **CRUD 전 과정**: 변경과 무관한 기존 CRUD도 정상인지
- **UI 렌더링**: 변경된 컴포넌트의 다른 props 조합 동작
- **연관 페이지**: 영향도 맵의 2차 연쇄 페이지 전체 확인
- **데이터 흐름**: 부모→자식 props 전달, 콜백 함수 연결

**체크리스트 FP-xx 결과 기록**. 테스트 후 데이터 원복 필수!

### 팀 3: 시나리오 폭격반 (general-purpose 에이전트)

**"다양한 상황에서도 버티는가?" — 체크리스트 SB-xx 담당**

시나리오 매트릭스에서 도출된 **최소 15개 시나리오** 실행:

**정상 시나리오 (5개+)**: 기본 동작, 다양한 유효 입력, 순차 작업, 데이터 조합
**경계값 시나리오 (5개+)**: 빈 값, 최소/최대, 특수문자, 날짜 경계, 긴 문자열
**비정상 시나리오 (5개+)**: 잘못된 타입, 없는 ID, 중복 요청, 권한 없는 접근

**각 시나리오별: 입력값 + 기대 결과 + 실제 결과 + 판정** 기록. 테스트 후 데이터 원복 필수!

### 팀 4: 연쇄반응 추적반 (general-purpose 에이전트)

**"수정이 다른 곳에서 폭발하지 않는가?" — 체크리스트 CR-xx 담당**

영향도 맵의 **모든 연쇄 경로**를 따라 테스트:
- **1차 연쇄**: 변경 파일을 import하는 모든 파일의 해당 기능 테스트
- **데이터 연쇄**: 같은 테이블을 쓰는 다른 API 정상 동작 확인
- **상태 연쇄**: 공유 상태(context, store)를 사용하는 다른 컴포넌트
- **이벤트 연쇄**: 변경된 이벤트 핸들러의 상위/하위 전파
- **캐시/메모이제이션**: useMemo, useCallback 의존성 변경 영향
- **라우팅**: 페이지 이동 시 변경된 상태 유지/초기화

**체크리스트 CR-xx 결과 기록**. 테스트 후 데이터 원복 필수!

### 팀 5: UI 브라우저 검증반 (Playwright 에이전트)

**"화면에서도 변경이 제대로 반영되는가?" — 체크리스트 BW-xx 담당**

`playwright-cli` 스킬로 **실제 브라우저**에서 변경된 기능의 UI 동작 검증:

- **변경된 페이지 접속**: navigate → 페이지 로드 → 콘솔 에러 0건 확인
- **변경된 기능 실행**: 버튼 클릭 → 모달 열림 → 폼 입력 → 저장 → 결과 확인
- **데이터 표시 정확성**: 화면에 표시된 값 (getText) == API 응답 값
- **기존 UI 정상**: 변경하지 않은 다른 페이지/기능도 정상 동작
- **모달 동작**: 열기(click) → 데이터 표시 → 수정 → 저장 → 닫힘 → 목록 갱신
- **에러 상태**: 잘못된 입력 시 에러 메시지 표시, 빈 데이터 시 빈 상태 UI

```
UI 검증 체크리스트:
  ☐ BW-01: 변경된 페이지 로드 정상 (콘솔 에러 0건)
  ☐ BW-02: 변경된 기능 CRUD UI 흐름 정상
  ☐ BW-03: 화면 데이터 == API 데이터 일치
  ☐ BW-04: 모달 열기/입력/저장/닫기 정상
  ☐ BW-05: 기존 페이지 UI 깨짐 없음
  ☐ BW-06: 에러/빈 상태 UI 표시 정상
각 항목: 스크린샷 증거 첨부
```

테스트 후 데이터 원복 필수!

## Phase 2: 결과 종합 + 교차 검증 + 재테스트

### 2-1. 체크리스트 결과 취합
```
━━━ 체크리스트 실행 결과 ━━━
[변경 정확성] 팀1: N/N PASS (XX%)
[기존 기능 보호] 팀2: N/N PASS (XX%)
[시나리오 폭격] 팀3: N/N PASS (XX%)
[연쇄반응 추적] 팀4: N/N PASS (XX%)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
전체: N/N PASS (XX%)
```

### 2-2. 교차 검증 (Cross-Validation)

팀 간 결과를 교차 대조:
- 팀1(코드 정상) vs 팀2(API 실패) → **런타임 전용 이슈** (빌드는 되지만 실행 시 에러)
- 팀2(API 성공) vs 팀3(경계값 실패) → **입력 검증 누락**
- 팀3(시나리오 통과) vs 팀4(연쇄 실패) → **사이드이펙트 미감지**
- 팀4(연쇄 정상) vs 팀1(코드 경고) → **잠재적 시한폭탄** (지금은 되지만 나중에 터짐)

### 2-3. FAIL 항목 영향도 역추적

```
❌ FAIL 발견 시 역추적 프로세스:
1. 원인 파일:줄번호 특정
2. 영향도 맵에서 해당 노드 찾기
3. 해당 노드와 연결된 모든 경로 확인
4. 같은 원인으로 영향받을 수 있는 다른 체크 항목 식별
5. 식별된 항목 추가 검증
```

### 2-4. FAIL 분류 + 수정 + 재테스트 (PDCA 반복)
```
🔴 Critical: 기능 자체가 작동 안 함 → 즉시 수정
🟡 Warning:  특정 조건에서만 문제 → 수정 후 재테스트
🟢 Info:     개선 권장 사항 → 기록만
```

수정 후:
1. 해당 시나리오 재테스트
2. **영향도 맵 연결 시나리오도 재테스트** (수정이 새로운 문제를 만들지 않았는지)
3. 빌드 확인
4. **최대 3회 반복**

## Phase 3: 크로스 체크 (2차 검증)

### 3-1. 빌드 + 린트
```bash
npx vite build  # 또는 npx next build
npx tsc --noEmit  # TypeScript 있을 경우
```

### 3-2. 변경 전후 비교
동일한 API 호출을 변경 전 코드(git stash)와 변경 후 코드로 실행:
- 응답 구조 동일 (새 필드 추가는 OK)
- 기존 필드 값 불변
- 응답 시간 비교

### 3-3. 사용자 시나리오 시뮬레이션
1. 로그인 → 메인 화면
2. 변경된 기능 사용
3. 다른 기능으로 이동
4. 다시 돌아와서 데이터 확인
5. 새로고침 후 데이터 유지 확인
